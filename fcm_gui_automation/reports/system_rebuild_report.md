# 프로그램 재현용 환경/설정/구조 보고서

작성일: 2026-04-27
대상 프로젝트: `D:\app\fcm_gui_automation`

## 1. 문서 목적

이 문서는 현재까지 만든 GUI 자동화 프로그램을 다른 환경에서 최대한 같은 형태로 다시 만들기 위해 필요한 내용을 정리한 보고서다.

이번 문서는 코드 한 줄 한 줄의 설명보다 아래 항목에 집중한다.

- 어떤 환경이 필요한가
- 어떤 폴더 구조로 프로젝트를 나누는가
- 어떤 설정 파일이 필요한가
- 어떤 실행 흐름으로 동작하는가
- 어떤 전제 조건이 있어야 안정적으로 재현되는가

## 2. 프로그램 개요

현재 프로그램은 Windows 데스크톱 GUI 자동화를 위한 최소 프레임워크다.
핵심 목적은 다음과 같다.

- 사용자가 실행할 대상 앱을 선택
- 사용자가 실행할 시나리오 YAML을 선택
- `pywinauto`를 통해 대상 창에 연결 또는 실행
- 입력, 클릭, 검증, 캡처, 종료를 시나리오 순서대로 수행
- 실패 시 로그와 스크린샷을 남김

즉, 이 프로젝트는 특정 앱 전용 스크립트라기보다 "Windows GUI 자동화 실행기"에 가깝다.

## 3. 권장 실행 환경

이 프로그램을 같은 방식으로 재현하려면 아래 환경이 필요하다.

- 운영체제: Windows
- Python: 3.12 계열
- GUI 자동화 라이브러리: `pywinauto`
- 대상 샘플 앱용 GUI 프레임워크: `PyQt6`
- 설정 파일 파싱: `PyYAML`
- PDF 및 이미지 처리: `Pillow`

현재 `requirements.txt` 기준 주요 패키지는 아래와 같다.

- `pywinauto>=0.6.8`
- `PyQt6>=6.7.0`
- `PyYAML>=6.0.2`
- `Pillow>=10.4.0`

## 4. 운영체제 의존 사항

이 프로젝트는 현재 사실상 Windows 전용이다.
이유는 다음과 같다.

- `pywinauto`가 Windows GUI 자동화를 전제로 한다.
- `UIA`와 `Win32` 계층을 활용하는 구조다.
- 파일 선택, 창 연결, 프로세스 제어가 Windows 환경에 맞춰 검증되었다.

따라서 Linux나 macOS에 그대로 이식하는 구조는 아니다.

## 5. 필수 전제 조건

이 프로그램을 같은 품질로 재현하려면 아래 조건이 충족되어야 한다.

- 자동화 대상 앱이 실제로 Windows 창을 생성해야 함
- 대상 앱이 표준 컨트롤 또는 UIA에서 어느 정도 식별 가능한 구조를 가져야 함
- 실행 계정이 대상 앱 실행과 UI 접근 권한을 가져야 함
- `comtypes` 캐시 생성이 가능한 쓰기 가능한 경로가 있어야 함

특히 `pywinauto + UIA` 조합에서는 `comtypes_cache` 관련 권한 이슈가 발생할 수 있으므로, 프로젝트 내부 캐시 경로 또는 사용자 영역 캐시 경로를 준비하는 것이 중요하다.

## 6. 프로젝트 루트 구조

현재 재현 대상 구조는 아래와 같다.

```text
fcm_gui_automation/
├─ main.py
├─ requirements.txt
├─ config.yaml
├─ scenarios/
├─ core/
├─ recognition/
├─ reports/
├─ tools/
├─ locators/
└─ .cache/
```

이 구조는 역할 분리가 명확하다는 장점이 있다.

## 7. 디렉터리별 역할

### 7-1. `main.py`

프로그램의 진입점이다.
전체 실행에서 아래 책임을 가진다.

- 설정 파일 로딩
- 앱 경로 선택 또는 CLI 인자 처리
- 시나리오 선택 또는 CLI 인자 처리
- 어댑터, 로거, 실패 처리기 조립
- 시나리오 실행 시작

### 7-2. `config.yaml`

자동화 엔진 수준의 설정 파일이다.
주요 항목은 다음과 같다.

- `backend`: 현재 `uia`
- `title_re`: 기존 창 연결 시 사용할 제목 패턴
- `python_command`: Python 스크립트 실행 시 사용할 명령
- `script_path`: 기본 앱 경로
- `start_timeout`, `focus_timeout`: 대기 시간
- 로그/스크린샷 저장 디렉터리

즉, `config.yaml`은 "엔진이 어떤 앱에 어떻게 붙을지"를 정의한다.

### 7-3. `scenarios/`

자동화 순서를 담는 시나리오 디렉터리다.
각 YAML 파일은 한 개의 테스트 케이스 또는 실행 절차를 의미한다.

현재 예시 시나리오의 기본 구성은 아래와 같다.

- 앱 실행 또는 연결
- 입력값 채우기
- 버튼 클릭
- 결과 텍스트 검증
- 스크린샷 저장
- 종료

즉, 이 폴더는 "무엇을 할 것인가"를 담는다.

### 7-4. `core/`

자동화 공통 실행 로직을 담는 계층이다.
대표 역할은 다음과 같다.

- 시나리오 step 실행
- YAML 로딩
- 실패 처리
- 로그 생성
- 스크린샷 파일 경로 생성

즉, `core`는 "흐름 제어와 운영"을 담당한다.

### 7-5. `recognition/`

실제 UI 인식 및 제어를 담당하는 계층이다.
현재는 `pywinauto_adapter.py`가 핵심 구현체다.

또한 향후 확장을 위해 아래 빈 구조가 잡혀 있다.

- `opencv_adapter.py`
- `ocr_adapter.py`
- `base.py`

즉, `recognition`은 "실제 화면/컨트롤과 상호작용하는 계층"이다.

### 7-6. `reports/`

실행 결과와 문서를 보관하는 폴더다.

- `logs/`: 자동화 실행 로그
- `screenshots/`: 성공/실패 캡처
- `*.md`, `*.pdf`: 각종 보고서

이 구조는 테스트 결과와 문서를 프로젝트 내부에서 함께 관리하게 해준다.

### 7-7. `tools/`

운영 보조 스크립트가 들어가는 폴더다.

- 환경 점검
- PDF 생성
- 향후 템플릿 수집

즉, 본 실행과 직접 연결되지 않는 유틸리티 모음이다.

### 7-8. `locators/`

현재는 자리만 잡혀 있는 디렉터리다.
추후 OpenCV 기반 이미지 매칭을 도입할 경우 템플릿 이미지가 들어가는 위치다.

### 7-9. `.cache/`

라이브러리 동작을 위한 로컬 캐시 디렉터리다.
특히 `comtypes` 관련 캐시를 프로젝트 내부에 둘 수 있도록 준비한 구조다.

## 8. 대상 앱 구조 전제

자동화 엔진만으로는 충분하지 않고, 대상 앱도 자동화 친화적으로 설계되어야 한다.
현재 샘플 대상 앱은 다음 구조를 가진다.

- `PyQt6` 기반 데스크톱 앱
- 여러 입력창과 버튼을 가진 단순 패널
- 각 위젯에 `objectName` 부여
- 상태 영역을 텍스트로 제공

자동화 재현을 위해 가장 중요한 대상 앱 조건은 다음과 같다.

- 입력창, 버튼, 상태창 등 핵심 컨트롤이 명확해야 함
- 가능하면 안정적인 식별자 이름을 가져야 함
- 버튼 클릭 후 결과를 확인할 수 있는 UI가 있어야 함

## 9. 설정 설계 원칙

이 프로그램을 다시 만들 때 중요한 설계 원칙은 "고정값을 가능한 한 코드 밖으로 빼는 것"이다.

현재 그 원칙은 아래처럼 반영되어 있다.

- 앱 경로: CLI 또는 파일 선택
- 시나리오 경로: CLI 또는 파일 선택
- 엔진 설정: `config.yaml`
- 실행 순서: `scenarios/*.yaml`

즉, 코드를 수정하지 않고도 실행 대상과 절차를 교체할 수 있게 설계되어 있다.

## 10. 실행 흐름 구조

현재 표준 실행 흐름은 다음과 같다.

1. 프로그램 시작
2. 환경 변수 및 캐시 경로 준비
3. 설정 파일 로딩
4. 실행 대상 앱 경로 결정
5. 시나리오 파일 경로 결정
6. 로그 시스템 구성
7. 자동화 어댑터 생성
8. 시나리오 step 순차 실행
9. 성공 시 로그/스크린샷 저장
10. 실패 시 스크린샷 저장 후 안전 종료

이 흐름은 단순하지만 테스트 자동화 POC로는 충분히 안정적이다.

## 11. 현재 action 체계

현재 재현 대상 시스템은 시나리오 기반 action 구조를 사용한다.
즉, YAML step마다 하나의 `action` 이름이 있고, 실행기가 그 이름을 해석한다.

현재 지원 action은 아래와 같다.

- `launch_or_connect`
- `set_text`
- `click`
- `verify_text`
- `screenshot`
- `safe_close`

이 체계는 재현 시 반드시 유지해야 할 핵심 구조 중 하나다.

## 12. 경로 선택 기능의 의미

최근 반영된 재사용성 개선에서 중요한 포인트는 "앱 경로와 시나리오 경로를 고정하지 않는다"는 점이다.

이 설계의 장점은 아래와 같다.

- 다른 앱으로 쉽게 전환 가능
- 여러 시나리오를 손쉽게 교체 가능
- 테스트 운영자가 코드를 몰라도 파일 선택으로 실행 가능
- CLI 기반 자동 실행과 수동 선택 실행을 둘 다 지원 가능

즉, 재현 시 이 기능은 권장 기능이 아니라 핵심 기능에 가깝다.

## 13. 창 연결 전략

이 프로그램을 동일하게 재현하려면 창 연결 전략도 중요하다.
현재는 두 가지 경우를 나눠 처리한다.

- 기존 창이 있으면 `title_re` 기반 연결 시도
- 새로 실행한 경우에는 `top_window()`로 최상위 창 확보

이 구조는 특정 제목에만 의존하는 것보다 범용성이 높다.

## 14. 로그와 증적 관리

재현 가능한 자동화 시스템을 만들 때는 실행 성공 자체보다 "무슨 일이 있었는지 남는가"가 중요하다.
현재 프로젝트는 이를 위해 아래 구조를 가진다.

- 콘솔 로그 출력
- `reports/logs/automation.log` 파일 저장
- `reports/screenshots/` 이미지 저장
- 별도 Markdown/PDF 보고서 저장

즉, 실행 흔적이 프로젝트 내부에 축적되는 구조다.

## 15. 권장 재현 순서

같은 시스템을 새로 만들려는 사람이 따라야 할 순서는 아래와 같다.

1. Windows + Python 환경 준비

3. 프로젝트 디렉터리 구조 생성
4. 대상 샘플 앱 준비
5. `config.yaml` 작성
6. 기본 시나리오 YAML 작성
7. `core` 계층 작성
8. `recognition` 계층 작성
9. 로그/스크린샷 폴더 생성
10. 환경 점검 스크립트 준비
11. 실제 시나리오 실행 검증

## 16. 꼭 포함해야 하는 파일 유형

동일한 프로그램을 다시 만들기 위해서는 최소한 아래 유형의 파일이 필요하다.

- 실행 진입점 파일
- 설정 파일
- 시나리오 YAML 파일
- UI 제어 어댑터 파일
- 실패 처리 파일
- 로그 설정 파일
- 스크린샷 경로 유틸
- 환경 점검 유틸
- 보고서 생성 유틸

즉, 자동화 엔진, 설정, 시나리오, 운영 도구의 4축이 있어야 한다.

## 17. 확장 가능한 구조로 만들기 위한 원칙

재현 시 아래 원칙을 유지하면 이후 확장이 쉬워진다.

- UI 제어 로직과 시나리오 해석 로직을 분리
- 설정값을 코드에 하드코딩하지 않기
- 결과 저장 위치를 프로젝트 내부에 통일
- 인식 방식별 어댑터 자리를 미리 분리
- 실패 처리를 독립 모듈로 유지

이 원칙 덕분에 나중에 OpenCV, OCR, 추가 action을 붙일 때 구조가 무너지지 않는다.

## 18. 향후 재현 시 고려할 추가 항목

같은 시스템을 더 완성도 높게 만들려면 아래를 같이 고려하면 좋다.

- 최근 사용 경로 저장
- 시나리오 다중 선택 또는 목록 UI
- YAML schema 검증
- action 템플릿화
- 실행 리포트 자동 생성
- 컨트롤 탐색 도구 제공

## 19. 결론

현재까지 만든 프로그램을 다시 만들기 위해 가장 중요한 것은 코드 세부 구현보다 다음 네 가지를 이해하는 것이다.

- Windows 기반 자동화 환경
- 설정 파일과 시나리오 파일의 분리
- `core`와 `recognition`으로 나뉜 구조
- 실행 결과를 남기는 로그/스크린샷/보고서 체계

즉, 이 시스템은 "자동화 코드 몇 개"가 아니라, 환경, 설정, 시나리오, 실행기, 증적 관리가 결합된 작은 프레임워크로 보는 것이 가장 정확하다.
